## 📚 Guide Reference Order

When authoring or reviewing any day's content, consult the guides in this order:

### Step 0: Research modern technologies

**`TECHNOLOGY-RESEARCH-GUIDE.md`**
Read this BEFORE writing any new content. Establishes standards for researching and selecting current best-in-class tools. Covers: research process using MCP tools, technology selection checklist, modern vs traditional approach formatting, and current best practices by ecosystem (Python, JavaScript, infrastructure). Critical for ensuring curriculum teaches production-ready practices, not legacy approaches.

### Step 1: Understand the ordering principle

**`ACTION-FIRST-GUIDE.md`**
Read this first. It defines the fundamental sequence: code success before deep explanation. Every other guide assumes you understand this ordering. Covers Phase structure (Frame → First Wrench Turn → Bridge → Tighten the Bolts → Road Test), common ordering failures, and the implementation checklist.

### Step 2: Apply voice, tone, and pattern standards

**`WRITING-STYLE-GUIDE.md`**
The master reference for Layer 1's "Senior Engineer in the Garage" voice. Covers: the Mechanic's Workflow section-by-section, all pedagogical patterns (scaffold file standards, interview anchor blocks, failure-forward diagnostics, spaced review cadence, logbook prompt design), text formatting rules, and Layer 1-specific pitfalls.

### Step 3: Expand thin content to teaching quality

**`LANGUAGE-EXPANSION-GUIDE.md`**
How to take a thin first draft and expand it into a teaching artifact. Covers expansion targets in priority order: TODO blocks → concept explanations → broken-state diagnostics → interview anchors → logbook prompts → README close. Includes before/after transformation examples for every target.

### Step 4: Add analogies

**`ANALOGY-LIBRARY.md`**
Indexed analogies by concept category, with difficulty levels and interview usage notes. Category 0 (Mechanic's Domain) is required for every day's opening analogy. Other categories are indexed by the day they apply. Quick reference table at the end maps day number to recommended analogies.

### Step 5: Add visuals

**`VISUAL-ENHANCEMENT-GUIDE.md`**
When and how to use ASCII diagrams, Mermaid flowcharts, data structure evolution diagrams, and broken-state side-by-side comparisons. Includes per-day visual guide (Days 1–5 detailed, Day 26, Day 43), standards for ASCII art, and whiteboard diagram requirements for interview readiness.

### Step 6: Verify

**`QUALITY-CHECKLIST.md`**
60-item binary checklist organized by section. Covers structural completeness, scaffold file quality, learning method quality, voice and tone, concept explanation depth, Road Test rigor, and code standards. Includes scoring thresholds, critical items, Day 1-specific requirements, and common issue fixes.

## 🧭 Layer 1 vs Layer 2 Guide Differences

These Layer 1 guides are adapted from the Layer 2 Curriculum guides. The key differences:

| Dimension          | Layer 2 (comprehensive)                | Layer 1 (mechanic)                             |
| ------------------ | -------------------------------------- | ---------------------------------------------- |
| Voice              | "Senior engineer at the coffee shop"   | "Senior engineer in the garage"                |
| Teaching persona   | Friendly mentor, 250-word intros       | Experienced mechanic, 50-100 word analogies    |
| Primary artifact   | Chapter markdown files                 | Scaffold `.py` files with TODOs                |
| Expansion goal     | Comprehensive understanding of concept | Interview-ready explanation + build confidence |
| Chapter cross-refs | "Remember Chapter 7?"                  | "Remember Day 1?"                              |
| Logbook prompts    | Reflection on learning                 | Synthesis and counterfactuals                  |
| Success metric     | Learner understands deeply             | Learner builds AND explains in interview       |
| Analogy domain     | Any domain, rich variety               | Mechanic/Garage for day-openers                |
| Visual priority    | Concept maps, mermaid everywhere       | Broken-state ASCII, pipeline diagrams          |
